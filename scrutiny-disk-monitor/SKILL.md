# Scrutiny Disk Monitor

## What This Does

Monitors the health of all your hard drives and SSDs using S.M.A.R.T (Self-Monitoring, Analysis, and Reporting Technology) data. Deploys Scrutiny — a self-hosted web dashboard that collects S.M.A.R.T metrics on a schedule, tracks historical trends, and alerts you before a drive fails.

**Example:** "Monitor 4 drives, get a Telegram alert when any drive shows pre-failure attributes, view health history in a web dashboard."

## Quick Start (5 minutes)

### 1. Check Prerequisites

```bash
# Verify Docker is installed
docker --version || echo "Install Docker first: https://docs.docker.com/engine/install/"

# Verify smartmontools (for collector)
which smartctl || sudo apt-get install -y smartmontools

# List your drives
lsblk -d -o NAME,SIZE,MODEL,SERIAL
```

### 2. Deploy Scrutiny (All-in-One)

```bash
# Create data directory
sudo mkdir -p /opt/scrutiny/{config,influxdb}

# Deploy with Docker
bash scripts/deploy.sh

# Access dashboard at http://localhost:8080
echo "Dashboard: http://$(hostname -I | awk '{print $1}'):8080"
```

### 3. Run First Scan

```bash
# Trigger immediate S.M.A.R.T scan
bash scripts/scan-now.sh

# View results in terminal
bash scripts/health-check.sh
```

```bash
bash scripts/deploy.sh --telegram-token "BOT_TOKEN" --telegram-chat "CHAT_ID"
```

**Use case:** Only monitor certain drives


### Workflow 4: Set Up Alerts

**Use case:** Get notified before a drive fails

```bash
# Telegram alerts
bash scripts/configure-alerts.sh --telegram \
  --token "$TELEGRAM_BOT_TOKEN" \
  --chat "$TELEGRAM_CHAT_ID"

# Email alerts
bash scripts/configure-alerts.sh --email \
  --smtp-host smtp.gmail.com \
  --smtp-port 587 \
  --smtp-user you@gmail.com \
  --smtp-pass "app-password" \
  --to admin@example.com

# Webhook alerts (Slack, Discord, etc.)
bash scripts/configure-alerts.sh --webhook \
  --url "https://hooks.slack.com/services/..."
```

### Workflow 5: Schedule Regular Scans

**Use case:** Automatically collect S.M.A.R.T data

```bash
# Set scan interval (default: every 6 hours)
bash scripts/configure.sh --scan-interval 21600

# Or use cron for more control
# Add to crontab: scan every 4 hours
echo "0 */4 * * * /opt/scrutiny/scripts/scan-now.sh >> /var/log/scrutiny-scan.log 2>&1" | sudo tee -a /etc/cron.d/scrutiny
```

## Configuration

### Docker Compose File

```yaml
# Generated by deploy.sh at /opt/scrutiny/docker-compose.yml
version: "3.8"
services:
  scrutiny:
    image: ghcr.io/analogj/scrutiny:master-omnibus
    container_name: scrutiny
    restart: unless-stopped
    cap_add:
      - SYS_RAWIO
      - SYS_ADMIN
    ports:
      - "8080:8080"
      - "8086:8086"  # InfluxDB
    volumes:
      - /opt/scrutiny/config:/opt/scrutiny/config
      - /opt/scrutiny/influxdb:/opt/scrutiny/influxdb
      - /run/udev:/run/udev:ro
    devices:
      # All block devices — auto-detected by deploy.sh
      - /dev/sda
      - /dev/sdb
    environment:
      - SCRUTINY_WEB_INFLUXDB_HOST=0.0.0.0
```

### Scrutiny Config (YAML)

```yaml
# /opt/scrutiny/config/scrutiny.yaml
version: 1

web:
  listen:
    port: 8080
    host: 0.0.0.0
  influxdb:
    port: 8086

notify:
  urls:
    - "telegram://BOT_TOKEN@telegram?channels=CHAT_ID"
    # - "smtp://user:pass@smtp.gmail.com:587/?to=admin@example.com"
    # - "slack://hook.slack.com/services/T.../B.../..."

devices:
  # Override detected device types if needed
  # - device: /dev/sda
  #   type: sat  # Force device type (sat, nvme, scsi, ata)
```

## Troubleshooting

### Issue: "Permission denied" accessing drives

**Fix:**
```bash
# Ensure Docker has raw I/O capability
# The deploy script adds SYS_RAWIO automatically
# If still failing, run collector as root:
sudo bash scripts/scan-now.sh
```

### Issue: NVMe drives not detected

**Fix:**
```bash
# Install nvme-cli
sudo apt-get install -y nvme-cli

# Verify NVMe is accessible
sudo nvme list

# Add to Docker devices explicitly
bash scripts/configure.sh --add-device /dev/nvme0n1
```

### Issue: "Device type not supported"

**Fix:**
```bash
# Override device type in config
bash scripts/configure.sh --device /dev/sda --type sat
# Types: sat, nvme, scsi, ata, auto
```

### Issue: Alerts not sending

**Check:**
1. Test notification URL: `bash scripts/test-alert.sh`
2. Verify config: `cat /opt/scrutiny/config/scrutiny.yaml`
3. Check logs: `docker logs scrutiny 2>&1 | tail -20`

## Key Principles

1. **Early warning** — Detect failing drives before data loss
2. **Historical tracking** — Trend S.M.A.R.T attributes over time
3. **Low overhead** — Scans run periodically, minimal resource usage
4. **Multi-drive** — Monitor HDDs, SSDs, and NVMe simultaneously
5. **Alert once** — Notify on status change, not every scan
